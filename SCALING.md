# SCALING — מ-מלון בודד ל-100 מלונות × 1000 אורחים

> מסמך הנדסי כן: מה נבנה עכשיו ועובד, מה המערכת מסוגלת להחזיק, ומה עוד
> *חייב* תשתית חיצונית כדי להגיע ליעד. בלי הבטחות שווא.

## 1. מה נבנה בסבב הזה (עובד ונבדק)

| רכיב | קובץ | מה הוא נותן |
|---|---|---|
| **נעילה per-guest** | `concurrency.js` (`withLock`) | שתי הודעות של אותו אורח מעובדות בזו אחר זו — אין דריסת מצב, אין כפילויות, אין אובדן. אורחים/מלונות שונים רצים במקביל. |
| **סמפור ל-AI** | `concurrency.js` (`createSemaphore`) | תקרת מקביליות לקריאות Anthropic (ברירת מחדל 24). 1000 אורחים לא פותחים 1000 חיבורים. |
| **retry + backoff + timeout** | `concurrency.js` | קריאות Google Places עטופות ב-timeout קשיח + 3 ניסיונות עם backoff על תקלה חולפת. שירות איטי לא תוקע ולא מפיל. |
| **rate limit per-guest** | `concurrency.js` (`createRateLimiter`) | בולם הצפה/abuse ממספר בודד. |
| **זהות מלון בכל הזרימה** | `tenant.js` (AsyncLocalStorage) | כל הודעה רצה תחת `hotelId` (מ-To של Twilio). כל קריאה פנימית יודעת לאיזה מלון היא שייכת — בלי לחווט פרמטר דרך 54 קריאות. |
| **בידוד state בין מלונות** | `state.js`, `checkin.js` | סשנים והזמנות ממופתחים לפי `(hotelId, phone)` / נושאים `hotelId`. אותו מספר טלפון בשני מלונות = שני סשנים נפרדים. |
| **מיפוי מספר → מלון** | `tenant.js` + טבלת `hotel_numbers` | כל מלון והמספר שלו. תשובות יוצאות מהמספר של המלון הנכון. |
| **בדיקות עומס ובידוד** | `scale.test.mjs` | 300 הודעות במקביל בשני מלונות — בלי קריסה, בלי דליפה, בלי אובדן. |

**כל 225 הבדיקות עוברות** (`npm test`).

## 2. מה המערכת מחזיקה **עכשיו** (תהליך בודד + SQLite)

- **מלונות:** רבים — הבידוד הלוגי מלא. אין תקרה ארכיטקטונית על מספר המלונות בתהליך.
- **אורחים במקביל:** מאות–אלפים בו-זמנית בתהליך בודד (Node async I/O). צוואר
  הבקבוק האמיתי הוא **קצב ה-API של Anthropic**, לא ה-CPU/הזיכרון שלנו — לכן הסמפור.
- **עמידות:** אף שגיאה לא מפילה את התהליך (`unhandledRejection`/`uncaughtException`
  נתפסים; כל הודעה עטופה ב-try/catch עם מענה גיבוי + הסלמה).
- **התאוששות:** SQLite עם WAL, write-through, הידרציה בעלייה. שורד ריסטארט.

### הגבול הקשיח של המצב הנוכחי
- **תהליך בודד.** ה-state (סשנים, נעילות, סמפור) חי ב-RAM של תהליך אחד. אפשר
  להריץ תהליך אחד גדול, אבל **לא כמה תהליכים/מכונות** שחולקים אורחים — שני
  תהליכים לא יראו את אותה נעילה per-guest ולא את אותו cache.
- **SQLite = כותב יחיד.** מצוין למלון-שניים ולתהליך בודד; לא מיועד לכמה כותבים
  מקביליים ממכונות שונות.

**מסקנה:** להדגמה, פיילוט, וכמה מלונות ראשונים — המצב הנוכחי מספיק ויציב.
ל-100 מלונות × 1000 אורחים בפרודקשן — צריך את סעיף 3.

## 3. מה נדרש כדי להגיע ל-100 × 1000 (תשתית חיצונית)

### 3.1 מסד נתונים מנוהל — PostgreSQL
**למה:** SQLite הוא כותב-יחיד ומקומי לתהליך. לריבוי מכונות צריך DB מרכזי
שתומך בכתיבות מקביליות, גיבוי, ו-replication.

**נקודת ההחלפה:** כל גישות ה-DB עוברות דרך *handle יחיד* שמוגדר ב-`db.js`,
והסכימה מוגדרת שם בלבד. הפונקציות שמריצות SQL בפועל (prepared statements)
יושבות ב-`state.js`, `checkin.js`, `config.js`, `tenant.js`, `idverify/registry.js`.
**המעבר ל-Postgres = החלפת המימוש בקבצים האלה** (אותן שאילתות, `pg` במקום
`node:sqlite`). הסכימה כבר מולטי-טננט (עמודת `hotel_id` בכל טבלה) ועם אינדקסים.

⚠️ **הבהרה כנה:** `node:sqlite` סינכרוני; `pg` אסינכרוני. המעבר דורש להפוך את
פונקציות ה-state ל-`async` (ואת הקוראים שלהן ל-`await`). זו עבודה אמיתית
(הערכה: 2–4 ימי פיתוח + בדיקות), לא "החלפת שורה". בנינו את הגבול נכון כדי
שזה יהיה מרוכז — אבל זה לא טריוויאלי, ואני לא מתיימר שכן.

**עלות (מנוהל, הערכה):** Postgres מנוהל (Railway/Render/Supabase/RDS) —
**~$25–70/חודש** להתחלה; גדל עם הנפח. AWS RDS/Aurora בסקייל — מאות דולרים/חודש.

### 3.2 Redis — סשנים ונעילות מבוזרות
**למה:** כדי להריץ **כמה מכונות** מאחורי load balancer, ה-cache של הסשנים
והנעילה per-guest חייבים להיות **משותפים** בין המכונות. Redis הוא הסטנדרט:
- סשנים: `GET/SET` לפי `hotelId:phone` (במקום ה-cache בזיכרון ב-`state.js`).
- נעילה per-guest מבוזרת: Redlock (במקום `withLock` המקומי).
- סמפור/rate-limit גלובליים: מונים ב-Redis.

**נקודת ההחלפה:** `state.js` (ה-cache) ו-`concurrency.js` (נעילה/סמפור).
הממשק זהה — `withLock`, `getSession` — כך שהקוראים לא משתנים.

**עלות:** Redis מנוהל — **~$10–50/חודש** להתחלה.

### 3.3 כמה עותקי אפליקציה + Load Balancer
- 2+ עותקי Node מאחורי LB (Railway/Render/Fly/ECS). ה-webhook של Twilio
  stateless — כל בקשה נושאת `To` (המלון) ו-`From` (האורח), וכל תהליך פותר
  את הזהות לבד. אחרי Redis (3.2), כל עותק רואה את אותו state.
- **עלות:** 2–4 עותקים — **~$40–200/חודש** בהתחלה.

### 3.4 Twilio — מספר לכל מלון
**מוכן בקוד:** טבלת `hotel_numbers` ממפה מספר נכנס → `hotelId`, ותשובות
יוצאות מהמספר של המלון (`fromNumberFor`). מה שצריך **תפעולית**:
- מספר WhatsApp Business לכל מלון (או Sender מאושר תחת מספר משותף +
  Messaging Service). כל מלון עובר אימות WhatsApp/Meta (יום–ימים לכל מלון).
- **עלות Twilio:** מספר ~$1–2/חודש; הודעות WhatsApp ~$0.005–0.08 להודעה לפי
  מדינה + תמחור conversations של Meta. ל-100 מלונות × אלפי הודעות/חודש —
  **מאות–אלפי דולרים/חודש**, תלוי נפח ומדינה.

### 3.5 אחסון מסמכי זיהוי — vault/object storage
ראה `SECURITY.md`. בפרודקשן התמונות עוברות ל-**S3+KMS / vault / PMS**
במקום דיסק מקומי. נקודת ההחלפה: `idverify/index.js`.
**עלות:** S3+KMS — זניח בהתחלה (~$5–20/חודש).

### 3.6 ניטור ולוגים
כרגע `console`. לפרודקשן: logging מובנה (pino) + שילוח (Datadog/Grafana/
CloudWatch) + alerting. **עלות:** ~$0–100/חודש לפי כלי ונפח.

## 5. סדר מומלץ (roadmap)

1. **עכשיו:** המצב הנוכחי מספיק לפיילוט של כמה מלונות בתהליך בודד.
2. **לפני צמיחה:** Postgres (3.1) — הכי חשוב; מסיר את גבול הכותב-היחיד.
3. **לריבוי מכונות:** Redis (3.2) + עותקים מרובים (3.3).
4. **לכל מלון חדש:** מספר Twilio + `registerHotelNumber` + `updateConfigFor`.
5. **תמיד:** ניטור (3.6) ו-vault למסמכי זיהוי (3.5) לפני אורחים אמיתיים.
